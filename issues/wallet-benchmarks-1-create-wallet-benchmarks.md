# Bounty: Reproducible Wallet Performance Test Harness

**Tier:** L — 150,000 XTM

## Description

Build a reproducible test harness that measures `minotari-cli` wallet performance across three modes: old wallet (`minotari_console_wallet` via gRPC), new wallet (`minotari-cli` library with offline signing), and payment processor (batch 1-to-many transactions via `minotari-cli`).

Implementation approach is up to you — Rust, Python, shell scripts, or any combination — as long as the results are reproducible and verifiable by a third party running the same harness against the same network.

## Deliverables

1. **Harness source code** in a new repo, buildable and runnable by anyone with access to a funded wallet on Esmeralda testnet.
2. **Run instructions** — complete, step-by-step, no implicit knowledge.
3. **Baseline result profile** — a structured report generated by running the harness, committed alongside the code, serving as proof the harness works and as a reference point for future comparisons.

## Wallet Modes

1. **Old Wallet (`minotari_console_wallet`)** — harness manages the wallet process lifecycle (spawn, wait for ready, run tests, tear down) and interacts via gRPC.
2. **New Wallet (`minotari-cli` library)** — uses the `minotari` crate directly: local UTXO selection, `sign_locked_transaction`, broadcast via HTTP RPC to a base node. No external wallet process.
3. **Payment Processor** — uses the new wallet's batch transaction capability to send 1-to-many transactions.

## Configuration Parameters

The harness MUST expose (and the result profile MUST record) the following parameters. Values below are the reference defaults for the baseline result profile.

| Parameter | Default | Notes |
|---|---|---|
| `A_fund` | 10,000 tXTM | Single funding UTXO per mode. ~71% of one Esmeralda block reward. |
| `C_min` | 3 | Confirmation depth before a UTXO is treated as spendable. |
| `volume_target` | 512 UTXOs | Target UTXO count after S1. |
| `doubling_rounds` | 6 | Take 1 → 64 UTXOs. |
| `fanout_outputs_per_tx` | 8 | Take 64 → 512 UTXOs in one round of 64 txs. |
| `concurrent_batches` | 8, 16, 32, 64, 128 | Ramp for S4. |
| `S4_T_budget` | 15 min | Stop condition per batch in S4. |
| `S5_M` | 100 | Total recipients paid in S5. |
| `S5_K` | 10 | Outputs per batch tx in S5 (→ 10 batch txs vs. 100 individual txs). |
| `fee_rate` | pinned | Explicit fee rate passed to all modes. Record the value used. |
| Wallet binary versions | pinned | Exact commit/tag of `minotari_console_wallet`, `minotari-cli`, base node. |

**Seed strategy:** fresh seed per wallet mode. Funding the next mode's seed can come from anywhere (faucet, prior mode sweep-back, miner wallet) — the funding step is not part of the measurement.

## Test Protocol

Nine scenarios run per wallet mode, in the order below. Every scan scenario starts with a wiped wallet data dir. Every scenario records: wall-clock duration, fees paid, success/failure counts with reasons, and final balance delta vs. expected.
When you need to rescan from genesis, use the api to change the seed words so that the encoded birthday reflects 0.

### B0 — Baseline Scan (empty wallet)

Floor cost of block-walk + view-key check with nothing to store.

- **Preconditions:** fresh seed, never funded. Base node synced to tip. Birthday = 0.
- **Steps:** launch scan from genesis → wait for wallet height == tip.
- **Verification:** 0 UTXOs, 0 balance, 0 outputs found.
- **Metrics:** `T_scan`, `blocks_per_sec`, `H_tip_start`, `H_tip_end`, peak RSS, peak CPU%.

### S0 — Funding Baseline

Identical starting state for all downstream scenarios.

- **Preconditions:** fresh seed, fresh wallet data dir. Base node synced.
- **Steps:** init wallet → record birthday `H_birth` → receive 1 UTXO of `A_fund` → wait for confirmation at depth ≥ `C_min`.
- **Verification:** exactly 1 spendable UTXO; balance == `A_fund`.
- **Metrics:** broadcast→mempool time; broadcast→confirmed-at-depth time; `H_birth`, tip height.

### S1 — UTXO Build-up (doubling + fan-out → 512)

Produces on-chain history for scan scenarios AND measures per-tx cost.

**Doubling phase (6 rounds):**

| Round | Txs (serial) | Inputs/tx | Outputs/tx | UTXOs after |
|---|---|---|---|---|
| 1 | 1 | 1 | 2 | 2 |
| 2 | 2 | 1 | 2 | 4 |
| 3 | 4 | 1 | 2 | 8 |
| 4 | 8 | 1 | 2 | 16 |
| 5 | 16 | 1 | 2 | 32 |
| 6 | 32 | 1 | 2 | 64 |

**Fan-out phase (1 round):** 64 txs, each 1 input → 8 outputs → 512 UTXOs.

**Per-round procedure:**
1. Snapshot balance and UTXO set.
2. Construct + broadcast all round txs serially.
3. Wait until all round txs reach depth ≥ `C_min`.
4. Refresh wallet state (tip refresh, not full scan).
5. Verify UTXO count matches target; verify `balance_after == balance_before − Σ round_fees`.

**Failure halts the scenario** — doubling cannot continue if a round's txs don't all land.

**Metrics (per tx):** construction time, broadcast→mempool time, broadcast→confirmed time, fee paid, outcome. **Per round:** wall-clock, total fees, failures. **Scenario total:** cumulative fees, cumulative wall-clock, balance reconciliation delta.

### S2 — Scan from Genesis (checkpoint 1)

- **Preconditions:** S1 complete; wipe wallet data dir; birthday = 0.
- **Steps:** launch scan from genesis → wait for height == tip.
- **Verification:** 512 spendable UTXOs rediscovered; balance matches post-S1; all S1 txs in history.
- **Metrics:** same as B0, plus outputs-found count.
- **Delta:** `T_scan(S2) − T_scan(B0)` ≈ cost of discovering/storing 512 UTXOs and their spent-input matches.

### S3 — Scan from Birthday (checkpoint 1)

Identical to S2 except **birthday = `H_birth`**. `blocks_scanned = H_tip_end − H_birth`.

### S4 — Concurrent Construction

Measures what each wallet mode does under concurrent load. Locking, serialization, selection contention, and stalls are the signal — NOT failures to engineer around.

- **Preconditions:** S1 complete.
- **Steps:** for each `N_concurrent` ∈ {8, 16, 32, 64, 128}:
  1. Pick `N_concurrent` distinct recipient addresses (self-sends OK).
  2. Fire all construction+broadcast calls in parallel (harness-level threads/async).
  3. Record per-tx: submit time, construction-complete time, broadcast outcome, confirmation time, error string.
  4. Wait for all terminal states OR `T_budget` elapsed (then record partial results and continue).
- **No retry. No backoff. No UTXO pre-partitioning.**
- **Metrics:** per-tx durations; batch wall-clock; success rate; max observed serialization gap between consecutive construction-complete events (exposes internal serialization); any double-selection rejections.

### S5 — Payment Processor Throughput

Headline comparison: batch 1-to-N vs. individual 1-to-1 sends paying the **same `M` recipients**.

- **Preconditions:** S4 complete. Record and disclose current UTXO count / balance (post-S4 state; do not normalize).
- **Recipient set:** fixed list of `M = 100` addresses, used by both arms.

**Arm A — Batch:**
- Submit `M/K = 10` batch txs (1 input → `K = 10` outputs), back-to-back.
- Wait for all confirmed at depth ≥ `C_min`.
- Record `T_batch`.

**Arm B — Individual:**
- Submit `M = 100` single-output txs, back-to-back.
- Wait for all confirmed at depth ≥ `C_min`.
- Record `T_individual`.

**Metrics:**
- `T_batch`, `T_individual`
- **Throughput multiplier = `T_individual / T_batch`** (headline)
- Total fees per arm; fee-per-recipient per arm
- Blocks consumed per arm
- Per-tx success/failure counts

**Runs per mode:** batch arm for payment processor; individual arm for new wallet and old wallet (for context).

### S6 — Scan from Genesis (checkpoint 2)

Same shape as S2 but after S5. Expected history ≈ 127 txs (S1) + S4 txs + 110 txs (S5) ≈ 250+ txs, several hundred UTXOs to rediscover.

**Delta signals:** `T_scan(S6) − T_scan(S2)` = cost of scanning S4+S5 history. `T_scan(S6) / T_scan(B0)` = scan slowdown factor vs. empty baseline.

### S7 — Scan from Birthday (checkpoint 2)

Same shape as S3 but after S5.

## Cross-cutting Metrics

Captured in every scenario:
- **Fees paid** — per tx, per round, per scenario total.
- **Error rate** — success / rejection / stall / timeout, with error strings.
- **Balance reconciliation** — `expected_balance − observed_balance` after every scenario. Flag any non-zero delta.
- **Chain tip heights** — at scenario start and end (for drift disclosure across modes).

## Result Profile (output format)

Structured output (JSON preferred, CSV acceptable), committed alongside the harness code as proof-of-run and reference baseline.

**Must include:**
- Hardware / environment disclosure: CPU model, RAM, disk type, OS, network path to base node (local vs. remote).
- Pinned versions: `minotari_console_wallet`, `minotari-cli`, base node — commit hash or release tag.
- All configuration parameter values used for the run.
- Per-scenario, per-mode block of metrics as described above.
- Computed deltas (e.g., `T_scan(S2) − T_scan(B0)`, S5 throughput multiplier).

## Acceptance Criteria

- Harness runs against Esmeralda testnet with no manual intervention beyond initial wallet funding.
- All three modes implemented; all nine scenarios (B0, S0–S7) produce results per mode.
- All configuration parameters exposed and recorded in the output.
- A third party can clone the repo, build the harness, fund the wallets, and reproduce the test — instructions are clear and complete.
- A baseline result profile is committed alongside the harness code.
- Results are output in a structured format suitable for automated comparison.
- **Harness does not hide wallet pain** (see principle below).

## Principle — Harness Measures, Does Not Engineer Around Wallet Pain

UTXO locking, selection contention, stalls, and failed tx construction under concurrency are **part of what the harness measures**. Do not add harness-level retries, backoff, UTXO pre-partitioning, throttling, or serialization that hides how a wallet mode actually behaves under load.

If a wallet stalls for 10 minutes on a locked store, that stall is the result. If a wallet mode cannot complete a scenario, that is a reportable result — not a reason to tune the harness until it does. Timeouts should be generous (so stalled-but-progressing wallets get credit for completing), but the observed behavior must show up in the metrics.
